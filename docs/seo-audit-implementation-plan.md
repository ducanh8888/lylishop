# LyliShop Technical SEO Audit and Implementation Plan

Repository: `ducanh8888/lylishop`

Domain: `https://lylishop.online`

Audit date: 2026-06-29

Scope: Next.js 15 App Router, TypeScript, TailwindCSS, static product showcase. No WordPress, CMS, database, backend, cart, checkout, fake address, fake email, fake reviews, or fake business data should be introduced.

## 1. Executive Summary

LyliShop already has a solid technical SEO base:

- Next.js Metadata API is used.
- Canonical URLs are present on the main public routes.
- `robots.txt` and `sitemap.xml` are generated by App Router metadata routes.
- Product pages are statically generated from `lib/products.ts`.
- JSON-LD exists for Organization, WebSite, Product, FAQPage, BreadcrumbList, ItemList, CollectionPage, and BlogPosting.
- Images are local assets and rendered with `next/image`.
- The site has a clear brand, product collection, footer links, CTA sections, FAQ, and social links.

The highest-risk issue is trust/data integrity around reviews and ratings. `lib/products.ts` and `lib/testimonials.ts` contain rating, review count, and testimonial data, and `lib/schema.ts` emits `aggregateRating`. If these ratings/reviews are not verified real customer data, they should be removed from UI and Product structured data before further SEO work.

The second major issue is strategy alignment. The user requirement says "khong tao blog", but the current repository already contains `/blog`, `/blog/[slug]`, and `lib/blog.ts`, and sitemap includes blog URLs. Because these URLs may already be indexed, they should not be deleted blindly. The team should decide whether to keep them as existing static "cam nang" pages, migrate their useful content into non-blog sections, or noindex/redirect them after checking Google Search Console.

The third issue is topic cannibalization. Homepage, `/moc-khoa-len`, `/products`, and blog posts all target overlapping terms around "moc khoa len handmade". The implementation should define one intent per page:

- Homepage: brand/pillar page for LyliShop and "moc khoa len handmade cute".
- `/products`: product collection / commercial listing.
- `/moc-khoa-len`: informational collection page or keyword-focused category guide.
- Product detail pages: product-specific long-tail subpages.
- Blog routes: only keep if existing indexed content is intentionally retained.

## 6. Technical SEO Audit

### 6.1 Metadata

Status: WARNING

Findings:

- Global metadata is centralized in `lib/seo.ts` and used from `app/layout.tsx`.
- Homepage, product collection, keyword collection, blog, privacy, and terms define route metadata.
- Product pages use `generateMetadata`.
- `/products`, `/privacy`, and `/terms` define OpenGraph but do not define route-specific Twitter metadata, so Twitter tags inherit global homepage title/description.
- `keywords` are still present in global metadata and some routes. This is not harmful by itself, but modern SEO value is limited.

Recommended actions:

- Add route-specific Twitter metadata to all public routes with unique page intent.
- Keep canonical, title, description, OpenGraph, Twitter, and robots consistent through a helper.
- Avoid keyword stuffing in metadata.
- Optionally remove or reduce `keywords` if the team wants a cleaner metadata strategy.

### 6.2 Canonical

Status: PASS

Findings:

- Homepage canonical resolves to `https://lylishop.online`.
- `/products` canonical resolves to `https://lylishop.online/products`.
- `/moc-khoa-len` canonical resolves to `https://lylishop.online/moc-khoa-len`.
- Product pages canonicalize to their own product URLs.

Recommended actions:

- Keep canonical URLs stable.
- Do not change product slugs without redirect analysis.

### 6.3 Robots

Status: PASS

Findings:

- `app/robots.ts` allows all public pages.
- Sitemap is declared.
- Host is set to `SITE.url`.

Recommended actions:

- Keep `Allow: /`.
- If `/blog` is later removed from strategy, decide whether to keep indexed, noindex, or redirect at route level rather than blocking in robots.

### 6.4 Sitemap

Status: WARNING

Findings:

- `app/sitemap.ts` includes homepage, `/products`, `/moc-khoa-len`, `/blog`, `/privacy`, `/terms`, all product pages, and all blog posts.
- Sitemap uses `new Date()` for `lastModified`, which changes every build. This can be acceptable but is not semantically accurate if content has not changed.
- Sitemap includes blog URLs, conflicting with the no-blog requirement unless the team intentionally treats current `/blog` as legacy/static guide content.

Recommended actions:

- Keep only intentional indexable routes in sitemap.
- Add stable `updatedAt` fields to product/blog/content data if accurate timestamps are needed.
- Do not remove blog URLs before checking index status and traffic.

### 6.5 Status Codes and Redirects

Status: PASS

Findings:

- `middleware.ts` redirects `www.lylishop.online` to `lylishop.online` with 308.
- Main live routes return 200.
- Unknown route returns 404.

Recommended actions:

- Keep apex domain as canonical.
- If product slug is renamed, add 308 redirects from old URLs to new URLs.

### 6.6 URL Structure

Status: WARNING

Findings:

- Product URLs use `/products/[slug]`.
- `/moc-khoa-len` is a strong keyword URL.
- `/blog` exists, despite the no-blog strategic requirement.
- Some product slugs are English while visible names are Vietnamese, e.g. `crochet-bear-keychain` for a rabbit product.

Recommended actions:

- Do not change existing URLs immediately.
- If changing slugs for SEO, create redirects and validate impact in Google Search Console.
- Prefer stable Vietnamese/ASCII slugs for future products.

### 6.7 Breadcrumb

Status: PASS / WARNING

Findings:

- Breadcrumb UI and JSON-LD exist for `/products`, `/moc-khoa-len`, product detail, and blog.
- Product detail breadcrumb uses `Trang chu > Moc khoa len > Product`, while CTA also links back to `/products`. This is acceptable but could be clearer.

Recommended actions:

- Decide whether product detail parent should be `/products` or `/moc-khoa-len`.
- Align visible breadcrumb, JSON-LD breadcrumb, and internal link strategy.

## 20. Final Recommendation

Start with trust cleanup before adding more SEO content. If ratings and testimonials are not verified, remove them from UI and schema first. Then normalize metadata and sitemap. After that, improve product data structure and internal links.

The implementation should avoid:

- fake reviews
- fake aggregate ratings
- fake Google Business Profile data
- fake address
- fake email
- fake social links
- unsupported product claims
- deleting or changing indexed URLs without redirect/index analysis

This plan is designed so the engineering team can implement changes in small, reviewable pull requests.
